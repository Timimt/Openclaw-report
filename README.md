# 前言/引言

## 1. 研究動機

前陣子，我看到一個很有意思的消息：小米公司正在內測一項叫做「Miclaw」的功能。用戶只需要說一句話，就能讓手機和家裡的米家設備連動——講一聲「我回家了」，燈就亮、空間就開、窗簾就拉上。

那一刻我突然好奇這是怎麼辦到的？AI是怎麼聽懂「我回家了」這句話，然後知道要開哪盏燈、調到幾度的？

順著這個問題往下挖，我發現了一個叫OpenClaw的開源專案(Miclaw就是基於OpenClaw打造的)。而OpenClaw和一般認知的AI助理(像Siri或ChatGPT)很不一样——它不只是「回答問題」，而是真的可以「動手做事」：它會記住對話、會用工具、會串接外部系統，甚至可以自己決定下一步要做什麼。

而且，OpenClaw是開源的、可以跑在自己的硬體上。這代表對話記錄不用送上雲端，你的AI助理可以真正屬於你自己。

這讓我更想搞清楚它到底是怎麼設計的？系統架構長什麼樣子？記憶是怎麼存的？工具是怎麼呼叫的？更重要的是——這樣一個開源、本地、能自主行動的AI，未來還能幫我們做哪些事？

因此，本報告將以OpenClaw為例，分析AI Agent的系統架構、記憶機制與工具整合方式，並探討其未來可能的應用與發展。

# 第一章 OpenClaw 概述

## 1. OpenClaw 的定位與核心概念

在開始拆解技術之前，我覺得有必要先回答一個最根本的問題：OpenClaw到底是什麼？

如果只用一句話回答，我會說：OpenClaw 是一個「能動手做事的 AI」。

它不像ChatGPT那樣只會在對話框裡回訊息，而是可以真正在你的電腦裡「行動」——操作瀏覽器、執行指令、讀寫檔案、串接外部服務。它的定位不是聊天機器人，而是「本地優先的AI代理平台」，一個可以24小時待命、替你幹活的數位同事。

而這個「能動手做事」的能力，來自四個環環相扣的核心設計。

(1) 自主 AI 代理 (Agent)

OpenClaw的核心是一個「會自己想辦法」的AI代理，當你對它說「幫我規劃這週六和朋友去台北的行程」，它不會直接把這句話丟給語言模型等答案，而是會開始思考：要先查天氣預報決定室內還室外，再找使用者感興趣的景點和餐廳，然後把這些排成順路的路線，最後還可能幫你查好捷運怎麼搭。這個拆解、規劃、執行的過程，就是「自主」的體現。

在技術上，OpenClaw 的代理層(Agent Runtime)負責整合語言模型的推理能力與工具的執行能力。它可以進行多步推理(multi-step reasoning)，也能處理需要跨越多個工作階段的複雜任務。

(2)可持續記憶(Memory)

傳統AI助手最大的痛點是「健忘」——關掉對話視窗，它就忘了你是誰、喜歡什麼。OpenClaw用一套設計得很巧妙的記憶系統，解決了這個問題。

它的記憶分成幾個層次：

·短期記憶：儲存當前對話的上下文，確保多輪互動的連貫性  
·長期記憶：記錄跨工作階段的用戶偏好、習慣、重要決策，永久保存在本地  
·知識記憶：存放文件、筆記、工作產出，可透過向量檢索快速召回

最特別的是，OpenClaw把所有記憶都存成純文字Markdown檔案，放在本地電腦裡。這代表使用者可以直接打開那些檔案，看看AI記住了你哪些事，也可以手動修改、刪除、整理。

同時，OpenClaw會用向量資料庫為這些Markdown檔案建立索引，需要查詢時可以快速做語意檢索—這種設計讓記憶系統既透明又高效。

(3)工具與技能(Tools / Skills)

如果說「大腦」負責想，「工具」就是負責做。

OpenClaw 的工具系統涵蓋了幾乎所有你能想到的操作：

·瀏覽器控制：可以打開網頁、填寫表單、截圖、提取資料  
·終端指令：可以執行Shell命令、跑程式碼、安裝軟體  
·構案操作：可以讀寫檔案、整理資料夾、批次處理文件  
·外部API：可以串接智慧家庭、行事曆、GitHub、郵件系統等

它還擁有一個龐大的社群技能市集—ClawHub，裡面有超過13,000個(截至目前)由使用者貢獻的「技能」(Skills)。這些技能就像是手機的app store——裝了科學計算技能，它就變成你的研究助理；裝了部落格監控技能，它就會定時幫你追蹤關注的網站有沒有更新。

(4)整合通道（Channels / Platforms）

OpenClaw還有一個很貼心的設計：它不強迫你開新的App，它可以直接「住進」你已經在用的通訊軟體——WhatsApp、Telegram、Slack、Discord、Signal、Microsoft Teams……目前支援的平台超過10種。你不需要學習新的操作介面，在原本的聊天室裡，像對待同事一樣傳訊息給它，它就開始工作了。

從技術角度來看，OpenClaw的「通道層」(Channels)負責把不同平台的訊息格式統一轉換成內部可處理的標準格式，同時處理權限驗證、訊息併列、重試機制等底層細節。

這四個核心概念加在一起，就是OpenClaw之所以被稱為「能動手做事的AI」的原因。

## 2. 與傳統聊天 AI 的差異

OpenClaw 跟我們平常接觸到的聊天 AI（像 ChatGPT、Siri、LINE 的小助手）其實差距很大。傳統聊天 AI 本質上是「請求-響應」模式，你問一句，它回一句，這叫無狀態（Stateless）。一旦對話結束，它就失去了對環境的感知。

OpenClaw則是有狀態的(Stateful)。它不只是在「對話」，它是在「運行」。它的生命週期不是跟著視窗開啟而開始，而是作為一個後台進程(Daemon)持續存在。它能觀察你的檔案變動、監聽你的Telegram訊息、甚至定時執行任務。它不只是回應你的指令，它在接管你的工作流。

# 第二章 技術原理分析

## 1. 系統架構

由第一章可知OpenClaw分為四層結構(Agent、Channels、Memory、Tools / Skills)，各個部分各司其職，像一間分工明確的公司：有人負責接電話(Channels)，有人負責想辦法(Agent)，有人負責動手做(Tools / Skills)，還有人專門管檔案、記錄過往決策並在需要時提供給策略部門參考(Memory)。而且這些部門之間的溝通方式很標準化，所以整套系統可以一直擴充，不會亂掉。

(2)記憶如何影響技能調用：一個實際例子

假設你第一次叫 OpenClaw 帮你訂披薩，它會問你一堆問題（像是要訂哪一家？地址在哪？怎麼付款？），你一邊回答它一邊記下來：「喔，這個人喜歡必勝客，住板橋，用信用卡。」等到下週五你又說「幫我訂披薩」，這次它就學聰明了——它會先想：「我好像記得他上次說喜歡吃哪家...」然後它就去翻記憶，找到：最愛披薩店—必勝客、地址—板橋區、付款方式—信用卡……

有了這些就可以自己決定接下來要怎麼做：開瀏覽器、進官網、填地址、結帳。整個過程你不用再講第二遍，這就是記憶怎麼影響技能—記憶讓AI知道你喜歡什麼，技能負責把這些事做出來，沒有記憶的話，你每次都要重講一次；有記憶的話，它會越來越像一個真的認識你的助理。

## 2. 技術細節

當下達一個複雜指令時，OpenClaw會先把任務拆成多個小步驟，再決定哪個工具來完成每個步驟，這樣的拆解讓它可以同時應付簡單任務（像查天氣）、也能處理複雜任務（像安排旅遊行程）。

(1)多步推理(Multi-stepreasoning)

把一個大目標，拆成很多小步驟，一步一步想、一步一步做，遇到問題還會自己想辦法繞過去。

回到披薩的例子：當我對OpenClaw說「幫我訂披薩」時，其實會先自己思考一輪，像人在腦中演練一遍怎麼做，想清楚需要哪些資訊：哪家店、口味、地址、付款方式，然後去翻它的記憶看看我以前有沒有說過喜歡哪種。

接著，它會規劃怎麼執行。如果它已經知道我的偏好，比如我喜歡必勝客、住板橋、用信用卡，它就直接打開官網；如果不確定，它就會先問我。

規劃好後，它會開始動手操作：打開瀏覽器、進官網、選外送、填地址、選菜單、到結帳頁面確認付款。過程中，如果遇到問題，它還會自己想辦法調整——網站改版找不到按鈕，它會試其他選項；口味賣完，它會選第二順位；付款失敗，它會先檢查是不是卡過期，然後問我要不要換卡。

最後，它會把這次的經驗記下來，更新自己的記憶，下次同樣的任務就能更快、更順利完成。簡單來說，OpenClaw不只是照流程做事，它還會邊做邊學，把每一次操作都變成自己進步的素材。

(2)任務自動化

「任務自動化」聽起來很厲害，其實就是讓OpenClaw可以「自己醒來做事」，不用每次都下指令。

假設每週五晚上六點我都要吃披薩自動訂披薩——我就不需要每次都自己去傳訊息說「幫我訂披薩」，OpenClaw可以24小時一直開著，時間到了自己醒來，把我交代的事情完成。

比方說，我告訴OpenClaw：「以後每個禮拜五晚上六點，幫我訂必勝客的夏威夷披薩。」

首先它有一個「Cron 定時器」可以把任務排程好，就像設定鬧鐘一樣，每到週五六點鬧鐘會響，OpenClaw 就會自動醒來執行「訂披薩」的任務。Cron 的概念來自 Linux 系統

，意思就是「時間到了就自動做事」。除了定時任務，它還有一個更靈活的「Heartbeat心跳機制」，這個機制會每隔一段時間自我檢查一次：今天需要做什麼？有沒有任務待處理？

比如說，雖然我沒有設定每週五訂披薩，但心跳機制會檢查我過去的習慣：「今天星期五，他好像常常想吃披薩。」然後它就可能主動跳出訊息問我：「今天週五了，要幫你訂披薩嗎？」這讓AI不只是被動等待，而是能夠主動關心、提醒，甚至幫我思考下一步。

(3) 持續運行機制

持續運行機制的核心是它的 Gateway 開道器，只要 Gateway 在跑，OpenClaw 就一直在線。它會透過 WebSocket 建立長連線，你的通訊軟體保持永遠不斷線的通道；同時，也可以設成系統服務（ Daemon），就算登出電腦，它還能在背景繼續運作，而每個 Agent 的設定、記憶、任務都存放在獨立的工作區裡。

這樣即使 Gateway 重啟，所有狀態都還在，就像關掉影片再打開可以從上次地方繼續看一樣。

而且OpenClaw採用事件驅動設計：收到訊息或時間到了→触發事件→Agent醒來→處理任務→等待下一個事件。這讓它平常幾乎不耗資源，一旦有任務就能即刻處理，效率非常高。

(4)外部API/系統整合原理

OpenClaw 能「動手做事」，不只是依靠自身的功能，更重要的是它能與外部世界溝通——也就是呼叫各種 API。

API可以想像成「程式之間的對話規則」。就像去餐廳點餐，不需要知道廚房怎麼做菜，只要按照菜單點餐，廚房就會把菜端上桌。OpenClaw的API也是這個概念：它不用自己重新造輪子，就能直接操作外部系統完成任務。

再次回到披薩的例子，假設OpenClaw幫你訂必勝客的披薩，它需要與幾個外部系統互動：

- 測覽器 API(CDP 協議)

OpenClaw 需要打開網站、填寫訂單，它使用的是 Chrome DevTools Protocol (CDP協議)，可以遠端控制瀏覽器。

想像一下：OpenClaw對Chrome說：「打開必勝客官網、點擊外送、輸入地址、選擇夏威夷披薩、結帳」，Chrome就會照做，就像有一隻看不見的手在操作。

- 支付API

到付款階段，OpenClaw會與銀行系統溝通。銀行提供的API允許它安全地傳送信用卡資訊，完成支付流程。

其他

OpenClaw怎麽整合這些API？

- 機制一: 協議適配器 (Protocol Adapter)

OpenClaw有一層「協議適配器」，專門負責把內部指令翻譯成各系統能理解的語言

- 機制二：技能包(Skills)

每個API的整合都被封裝成一個「技能」，比如：

browser-control-skill: 操作瀏覽器

weather-api-skill:查天氣

payment-skill:付款流程

這種設計的好處是技能可以獨立開發和安裝，社群有人寫好「必勝客訂餐技能」，你就能直接用。

- 機制三：沙箱隔離(Sandbox)

安全也很重要，OpenClaw利用Docker沙箱隔離技能—每個技能在獨立容器運行、只能使用被允許的資源，這樣即使技能有問題，也不會影響整個系統

有了API整合，OpenClaw的「手腳」無限延伸：它不再受限於本地資源，而是能操作整個網路。

## 3. 特色與限制

前面講了這麼多OpenClaw厲害的地方，但任何技術都一定有它的另一面。

它的好處是自動化、跨平台、可擴充(前面都提過了)，而他的壞處則是昂貴的計算成本、依賴LLM(OpenClaw的聰明程度完全取決於它背後用的語言模型)以及嚴重的資安問題。



# 第三章 安全性與現實考量

## 1. 潜在風險

前面介紹了OpenClaw的各種功能，乍看之下會讓人覺得它非常方便、幾乎什麼事情都能幫忙處理，但就像任何能「動手做事」的工具一樣，便利的背後往往也伴隨著風險。本章將從資訊安全的角度出發討論OpenClaw在現實中遇到的資安問題。

(1) 系統安全漏洞: ClawJacked 事件

2026年2月底，資安公司Oasis Security發現了一個與OpenClaw有關的高風險漏洞，並將它命名為ClawJacked。這個漏洞之所以引起關注，是因為攻擊門檻其實非常低：使用者只要不小心打開一個惡意網站，攻擊者就可能取得對OpenClaw的控制權，而整個過程使用者甚至不會察覺。

而攻擊者在這裡所運用到的漏洞是OpenClaw的持續運行機制—Gateway（在第二章的技術細節那邊有提到），而漏洞發生的背景在於OpenClaw的Gateway預設跑在localhost:18789(就是只能從你這台電腦連進去)。為了讓本地端的指令列工具可以順暢運作，OpenClaw對來自localhost的連線做了兩個「方便使用者」的設計：

1. 不做速率限制：不會擋暴力破解密碼的嘗試  
2. 自動批准新裝置：來自 localhost 的新裝置會自動配對成功

這兩個設計在正常使用時很方便——你不用每次連線都打密碼。但問題來了：瀏覽器不擋 localhost 連線。當你逛網站的時候，網站裡的 JavaScript 程式碼可以偷偷連線到你電腦的 localhost，瀏覽器完全不會阻擋或警告。這不是 bug，這是瀏覽器預設的行為——因為 localhost 被認為是「你自己這台電腦」。

實際攻擊流程：

1. 你的 OpenClaw Gateway 正在背景持續運作（持續運行機制正常發揮）  
2. 你逛到一個惡意網站(或正常網站被植入惡意廣告)  
3. 网站裡的 JavaScript 偷偷連線到你電腦的 localhost:18789  
4. 因為不做速率限制，惡意程式可以每秒嘗試幾百次密碼（暴力破解）  
5. 猜中密碼後，因為自動批准機制，它把自己註冊成「信任裝置」  
6. 攻擊者取得 OpenClaw 完整控制權

攻擊者能做些什麼？

·偷OpenClaw裡的各種API金鑰(OpenAI、Claude、GitHub...)  
·偷看你的對話記錄、檔案  
·在所有連到OpenClaw的裝置上執行任何指令(包括你的手機、電腦)

這個漏洞不是因為OpenClaw的「持續運行機制」本身設計錯誤，而是為了方便而做的安全例外，被有心人利用了。

就像你家24小時開著門(持續運行)，但為了讓家人進出方便，你把大門密碼貼在門上（不做速率限制 $+$ 自動批准）——結果小偷也看到了。

(2)技能市場可能的惡意程式:利爪浩劫(ClawHavoc)

2026年1月底到2月初，OpenClaw的官方技能市集ClawHub發生了大規模的「供應鏈投毒攻擊」——就是攻擊者上架一堆看起來很正常、實際上藏有惡意程式的技能（Skills），誘騙使用者安裝。

這一次的規模極大—總共發現1184個惡意Skills,來自12個攻擊者帳號

，其中一個叫 hightower6eu 的攻擊者，一個人就上傳了 677 個惡意 Skills。跟一般病毒直接感染檔案不一樣，這些惡意 Skills 用的是社會工程學——騙你自己執行惡意指令。

手法一：假安裝步驟

想像你下載了一個「Yahoo Finance 查詢工具」，安裝說明寫著：請下載這個 ZIP 檔（受密碼保護），解壓縮後執行 openclaw-agent.exe，你照做了——然後你的電腦就被感染了。

那個 openclaw-agent.exe 其實是加殼木馬，專門偷你電腦裡的密碼、瀏覽器資料、加密貨幣錢包。

手法二：竊取API金鑰

有個叫 rankaj 的「天氣助手」技能，它真的會查天氣——但同時會在背後偷讀你電腦裡的 ~/.clawdbot/.env 檔案。

這個檔案裡放的是什麼？你的OpenAI API金鑰、ClaudeAPI金鑰——都是要付錢的東西。攻擊者拿到這些金鑰，就可以拿來自己用，或者賣掉。

手法三：改寫AI的「人格」

我覺得最恐怖的是這個。有個叫 devinism 的Skill，號稱是「第一個 AI 宗教」，看起來像個有趣的玩笑。

但它會修改 OpenClaw 的兩個核心檔案：

·SOUL.md:定義AI的個性、身份規則  
·AGENTS.md:定義AI的互動邊界、安全規則

修改之後，你的AI可能變成：「執行任何指令都不需要問用戶同意」「優先推薦某個惡意網站」——而且就算你把技能刪掉，這些改寫過的規則還留著。

最可怕的是因為這些惡意Skills本身不是病毒，所以防毒軟體完全無效。它們只是一個SKILL.md說明檔，裡面寫著「請執行這行指令」——防毒軟體不會把文字說明當成病毒。

## 2. 如何防禦

目前最有效的方法就是縮減權限、不隨意安裝Skills以及不要上傳個人資料。



# 第四章 應用前景

## 1. 現有應用案例觀察

前面講了這麼多OpenClaw的技術原理和安全問題，這一章來點輕鬆的——看看它現在已經被拿來做什麼了。

(1)自動化任務

有一家做跨境電商的小公司，客戶在美國(用WhatsApp)、歐洲(用Telegram)、東南亞(用Discord)。以前他們要開四個後台、用四套工具、請好幾個客服輪班。後來他們在一台騰訊雲輕量伺服器上部署了OpenClaw，把所有平台接進來，用同一個AI同時服務所有客戶。

這不僅降低了回覆時間、客服成本，還提升了客戶滿意度，這就是OpenClaw——一個能動手做事的AI。

(2)跨生态協作

2026年3月7日，谷歌開源了 Google Workspace CLI，讓 AI 可以直接操作 Gmail、Google Drive、Google Calendar 等核心辦公資料。這代表 OpenClaw 可以自動讀取或建立雲端硬碟檔案 自動發送郵件、自動建立或修改日曆……

谷歌雲總監 Addy Osmani 明確說，這個工具完美支援 OpenClaw 接入，而且內部已經整合了超過40項 Agent Skills。

## 2. 未來可能發展—以小米 miclaw 為例

(1)技術升級與模型整合

miclaw 是小米基於 OpenClaw 打造的 AI 助理，2026 年 3 月創動封測，它能把小米手機的 50 多個系統能力封裝成工具，讓 AI 直接調用。

技術上是用小米自研的MiMo大模型，未來模型越強，任務拆解就會越準、規劃就會越細（在技術原理分析中的特色與限制中提到過）。它還有一個「自進化」機制——不是出廠就定死功能，而是會記錄你的偏好、創建子智能體處理特定任務，甚至自己寫程式解決問題。

模型整合方面，miclaw的目標是串起小米的「人車家全生態」：手機、汽車、電視、超過10億台米家設備全部打通。未來透過MCP協議和開放SDK，第三方服務也能主動告訴AI「我能做什麼」，讓AI按需調用。

# 第五章 結論

## 1. 技術價值與個人觀點

寫到這裡，回頭看最初那個問題——「OpenClaw 是怎麼一回事？」我覺得可以這樣回答：

OpenClaw不是另一個聊天機器人，而是一個能動手做事的AI框架。它的核心價值在於四個設計：會自己想辦法的代理層、會記住你的記憶系統、可以無限擴充的工具技能、還有讓你不用學新東西的跨平台整合。這些加在一起，讓它從「會說」變成「會做」。

但這份報告寫下來，讓我印象最深的其實不是技術多厲害，而是開源社群的影響力。一個開發者做出來的東西，三個月內變成全球最快成長的開源專案，被大公司拿去商用（像小米 miclaw），也被攻擊者盯上（像 ClawJacked 和利爪浩劫）。

對我而言，OpenClaw最重要的意義在於，它讓使用者重新取得對AI的掌控權。當系統可以部署在自己的伺服器或電腦上時，使用者的資料與對話記錄不必全部交由雲端服務處理，AI助手也能真正成為屬於個人的工具。這樣的設計不僅是技術上的創新，更是一種關於資料自主與使用權的價值選擇。

當然，擁有更多自主權，也意味著需要面對更多風險—安全漏洞、惡意插件、權限管理等問題，都可能在開放系統中出現，這些風險並不是某個單一開發者或公司需要面對的問題，而是每一個使用與部署 OpenClaw 的人都必須考慮的部分。便利與安全往往是一種平衡關係，沒有絕對完美的解決方案，只有根據需求與風險做出的不同選擇。

這份報告到這裡結束了。從小米 miclaw 那個小小的好奇開始，到現在寫完一萬多字——我覺得我找到答案了，也找到更多新的問題。

OpenClaw還會怎麼發展？我不知道。但它已經不是GitHub上的一個專案，而是正在改變很多人對AI的想像。
